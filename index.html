Relatório

Introdução

Este relatório documenta o planejamento, a implementação e a homologação de uma Intranet corporativa integrada, desenvolvida em ambiente GNU/Linux. O objetivo central do experimento é consolidar os conhecimentos teóricos da camada de aplicação através da montagem de uma topologia de rede local funcional, configurando e integrando quatro serviços essenciais: DHCP, DNS, SMTP e WWW.

O projeto baseou-se na interconexão de estações e servidores para simular a infraestrutura de uma empresa fictícia. A gerência de endereçamento foi centralizada com o protocolo DHCP, configurado para automatizar a distribuição de IPs, máscaras e gateways. A resolução de nomes e a localização dos serviços dentro do domínio escolhido foram estabelecidas via DNS, implementando zonas de busca direta e reversa para traduzir FQDNs em endereços lógicos.

Para a comunicação e compartilhamento de dados, implantou-se o serviço de correio eletrônico utilizando o SMTP e protocolos de recuperação para o fluxo interno de e-mails. Por fim, o serviço WWW foi estruturado com o servidor Apache2 para hospedar a página principal da organização e os domínios virtuais dos membros do grupo, aplicando diretivas de restrição de acesso a diretórios.

Além da ativação dos serviços, o experimento englobou a configuração de tabelas de roteamento, regras de NAT para comunicação externa e a análise detalhada do tráfego de pacotes. Dessa forma, o trabalho demonstra de forma prática a interação cliente-servidor e o funcionamento dos protocolos que sustentam uma rede local privada.

Fundamentação Teórica

De acordo com Tanenbaum (2003), os serviços de camada de aplicação são a interface direta entre os programas de usuário e a infraestrutura de rede subjacente. Para o funcionamento de uma intranet corporativa, quatro pilares teóricos são fundamentais:

Simulação de DHCP

O DHCP automatiza a configuração da rede, eliminando a necessidade de designar manualmente endereços IP, máscaras de sub-rede e gateways para cada máquina. O protocolo opera em um modelo cliente-servidor e utiliza o processo conhecido como DORA (Discover, Offer, Request, Acknowledge). O servidor empresta os IPs por um tempo determinado que chamamos de lease time, podendo também fixar endereços lógicos baseados no endereço físico o MAC Address da placa de rede do cliente, garantindo consistência para máquinas que não podem mudar de IP.

Simulação de DNS

O DNS é um banco de dados distribuído e hierárquico cuja principal função é traduzir nomes de domínio humanamente legíveis, como FQDNs - Fully Qualified Domain Names para endereços IP e vice-versa, ou seja a resolução reversa. A infraestrutura depende de um resolver no lado do cliente que interroga os servidores. Um servidor pode ser Autoritativo, sendo assim, possui os arquivos de zona oficiais, como Master ou Slave ou Caching-only que apenas armazena respostas recentes para acelerar buscas. O serviço gerencia diversos tipos de registros, como A (endereço IPv4), PTR (ponteiro reverso) e MX (apontamento para o servidor de e-mails).

Simulação de SMTP

A arquitetura de e-mails divide-se em agentes de transferência (MTA) e agentes de entrega/usuário (MDA/MUA). Sendo assim, utilizamos os protocolos a seguir:

  • SMTP: É um protocolo de envio (push). Utilizado para transferir a mensagem do cliente para o servidor, ou entre servidores.
  • POP3/IMAP: São protocolos de recuperação (pull). Permitem que o cliente baixe ou sincronize as mensagens armazenadas no servidor. Historicamente restrito a textos simples (RFC 822), o correio eletrônico moderno utiliza o padrão MIME (Multipurpose Internet Mail Extensions) para permitir o anexo de arquivos binários, imagens e áudios. No servidor, os e-mails podem ser armazenados em formatos como mbox que é um único arquivo contendo todas as mensagens ou maildir que é um arquivo individual para cada mensagem

As estações clientes se conectam a este servidor por meio de um switch local, recebendo suas configurações via DHCP e utilizando o servidor como gateway e resolvedor DNS.

Roteiro de Instalação e Configuração

DHCP

Neste experimento foi implementado um servidor DHCP (Dynamic Host Configuration Protocol) em ambiente Linux com o objetivo de automatizar a configuração de rede das estações de trabalho. Inicialmente, foi configurada uma rede local com endereçamento estático para garantir a comunicação entre os equipamentos. Em seguida, o serviço DHCP foi configurado para distribuir dinamicamente endereços IP, máscara de sub-rede, gateway padrão e servidor DNS aos clientes da rede. A atividade permitiu observar o processo de concessão automática de endereços, simplificando a administração da rede e reduzindo a necessidade de configurações manuais em cada estação.

4.1.1. Configurar interfaces de rede

Inicialmente foi realizada a identificação das interfaces de rede disponíveis na máquina destinada a atuar como servidor DHCP. Para isso, foi utilizado o comando ip addr. O comando exibe todas as interfaces de rede presentes no sistema. Após a identificação da interface, foi realizada a configuração de um endereço IP estático no servidor. Para isso, foi editado o arquivo de configuração de rede:

sudo nano /etc/network/interfaces

A configuração foi alterada para:

auto enp0s3f6
iface enp0s3f6 inet static
address 192.168.10.10
netmask 255.255.255.0

Após a alteração do arquivo, as configurações foram aplicadas através dos comandos:

sudo ifdown enp0s3f6
sudo ifup enp0s3f6
4.1.2. Instalação e configuração do servidor DHCP

Após a preparação da interface de rede e definição do endereço IP estático do servidor, foi realizada a instalação do serviço DHCP. Para isso, utilizou-se o comando:

sudo apt install isc-dhcp-server

Concluída a instalação, foi realizada a edição do arquivo principal de configuração do serviço DHCP:

sudo nano /etc/dhcp/dhcpd.conf

Para o experimento, foi utilizada a seguinte configuração:

default-lease-time 600;
max-lease-time 7200;
authoritative;
option subnet-mask 255.255.255.0;
option broadcast-address 192.168.10.255;
option routers 192.168.10.10;
option domain-name-servers 192.168.10.10;
option domain-name "TerabitSolution.local";

subnet 192.168.10.0 netmask 255.255.255.0 {
    range 192.168.10.100 192.168.10.150;
}

host servidor_web {
    hardware ethernet <MAC_DO_CLIENTE>;
    fixed-address 192.168.10.20;
}

Após a configuração do arquivo, foi necessário informar ao serviço qual interface de rede seria utilizada para atender as requisições DHCP editando o arquivo /etc/default/isc-dhcp-server alterando o parâmetro INTERFACESv4="enp0s3f6". Por fim, o serviço foi iniciado através do comando:

sudo systemctl restart isc-dhcp-server
4.1.3. Configuração do cliente DHCP

Foi realizada a configuração da estação cliente para obtenção automática dos parâmetros de rede editando /etc/network/interfaces. A interface foi alterada para iface enp0s3f6 inet dhcp e o serviço religado. O comando dhclient enp0s3f6 é responsável por iniciar o processo de comunicação com o servidor DHCP. Por fim, foi realizado um teste de conectividade utilizando o comando ping 192.168.10.10.

4.1.4. Análise do processo de concessão do DHCP

Foi utilizado o software Wireshark, iniciado através do comando sudo wireshark. Após a execução, foi aplicado o filtro dhcp. Durante a captura realizada, foi possível observar o processo de negociação DORA (Discover, Offer, Request e Acknowledge).

4.1.5. Reserva de Endereço por MAC Address

Para implementar a reserva baseada em MAC, foi inserida a diretriz hardware ethernet e fixed-address para isolar a entrega estática, finalizando com o recomeço do serviço para aplicar as mudanças de estado.

DNS

O DNS (Domain Name System) é um serviço responsável pela resolução de nomes em redes TCP/IP, permitindo associar nomes de domínio a endereços IP. Neste experimento foi realizada a implementação de um servidor DNS em ambiente Linux utilizando o software BIND9 para a criação do domínio TerabitSolution.local e zonas de resolução direta e reversa.

4.2.1. Configuração do domínio

Foi verificada a configuração de nome atual (hostname) e ajustado o arquivo /etc/hostname. Em seguida, configurou-se /etc/hosts:

127.0.0.1       localhost
192.168.10.10   dns.TerabitSolution.local dns
192.168.10.20   mail.TerabitSolution.local mail
192.168.10.30   www.TerabitSolution.local www
4.2.2. Configuração do cliente DNS

Ajustamos o resolver editando o arquivo /etc/resolv.conf:

domain TerabitSolution.local
nameserver 192.168.10.10
4.2.3. Instalação e configuração do servidor DNS

A instalação foi realizada através do comando:

sudo apt install bind9 bind9utils bind9-doc

Foi realizada a edição do arquivo /etc/bind/named.conf.local onde foram adicionadas as definições das zonas direta e reversa do domínio. A zona direta (db.TerabitSolution) e zona reversa (db.reverse) foram geradas contendo os registros de SOA, NS, MX e ponteiros A/PTR correspondentes à topologia da rede. Finalizou-se com a validação (sudo named-checkconf) e o restart do serviço: sudo systemctl restart bind9.

4.2.4. Validação e teste do DNS

Testes de resolução direta, reversa e NSLookup foram feitos utilizando os utilitários de comando (host -d ..., nslookup ...), confirmando que os nomes configurados eram perfeitamente traduzidos pelo servidor BIND9.

SMTP

O SMTP (Simple Mail Transfer Protocol) é o protocolo padrão responsável pelo envio de mensagens eletrônicas em redes. Foi implementado usando Postfix para o transporte (push) das mensagens e Dovecot para a leitura e recuperação via POP3/IMAP (pull).

3.4.1. Instalação e Configuração do Servidor SMTP (Postfix)
sudo apt-get install postfix
sudo dpkg-reconfigure postfix

Estabelecimento da raiz mail.TerabitSolution.local para envios assíncronos das filas de emails, aceitando redes base na opção mynetworks (127.0.0.0/8, 192.168.10.0/24).

3.4.2. Criação de Usuários e Testes do Serviço SMTP

Os usuários locais foram criados através do comando sudo adduser mayara, o que possibilita caixas postais. Foi realizado o envio teste através do utilitário mail -v mayara@TerabitSolution.local e a caixa de correio checada no diretório /var/mail/mayara.

3.4.3. Configuração do Serviço POP3/IMAP (Dovecot)

Para a recuperação, foi instalado o Dovecot:

sudo apt install dovecot-pop3d dovecot-imapd

O arquivo /etc/dovecot/conf.d/10-mail.conf foi alterado para utilizar o armazenamento local no formato mbox (mail_location = mbox:~/mail:INBOX=/var/mail/%u). As portas 110 (POP3) e 143 (IMAP) foram confirmadas escutando com ss -tlnp.

WWW

O serviço WWW foi estruturado com o servidor Apache2, servindo desde as páginas padrão do domínio principal TerabitSolution.local até os domínios virtuais restritos aos funcionários do grupo.

4.3.1. Instalação do Apache2
sudo apt install apache2

Verificando os processos em execução via ps -aux | grep apache2 e testando localmente com http://192.168.10.10 no browser para visualizar a página de sucesso padrão.

4.3.2. Configuração Básica do Apache/HTTP

A página principal do grupo contendo os links base do laboratório foi colocada no diretório principal /var/www/html/index.html, acessível através de http://www.TerabitSolution.local.

4.3.3. Configuração de Diretórios Pessoais (public_html)

O módulo de pastas de usuário foi ativado no sistema:

sudo a2enmod userdir
sudo systemctl restart apache2
mkdir ~/public_html

Definindo as permissões restritas (755 e 644) para as pastas do root e disponibilizando as URLs na estrutura de til http://www.TerabitSolution.local/~nome.

4.3.4. Controle de Acesso Utilizando .htaccess

Foi garantida a não-exibição do index das pastas editando um arquivo .htaccess na raiz do diretório do usuário. A diretiva Options -Indexes impede o directory listing malicioso das contas.

4.3.5. Configuração de Domínios Virtuais

Foram configurados Virtual Hosts mapeando os nomes, por exemplo, mayara.TerabitSolution.local. Após apontar os registros na zona BIND9, os arquivos em /etc/apache2/sites-available/mayara.conf utilizaram ServerName e DocumentRoot /var/www/mayara, sendo ativados com sudo a2ensite mayara.conf.

4.3.6. Implementação de Server Side Includes (SSI)

O SSI foi utilizado como introdução ao processamento dinâmico no lado do servidor. O módulo include foi ativado (sudo a2enmod include) e nas configurações foi adicionado suporte aos arquivos `.shtml`. A tag <!--#echo var="DATE_LOCAL" --> foi utilizada para retornar a data da máquina ativamente antes de repassar o HTML na requisição GET.

Respostas às questões de cada roteiro

DHCP

Coloque um analisador de tráfego (Wireshark, tcpdump ou similar) de modo a identificar os tipos de diálogos existentes entre a estação de trabalho que está requisitando um endereço IP e o servidor DHCP. Anote a sequência de mensagens e apresente-as, associadas a uma pequena explicação sobre cada uma delas.
keyboard_arrow_down

Para analisar a comunicação entre cliente e servidor DHCP foi utilizado o software Wireshark. Após selecionar a interface de rede enp0s3f6 e aplicar o filtro dhcp, foi possível observar as mensagens trocadas durante o processo de obtenção automática de endereço IP.

Durante a captura foi identificado o processo de negociação DHCP, conhecido como DORA (Discover, Offer, Request e Acknowledge), composto pelas seguintes etapas:
- DHCP Discover: mensagem enviada pelo cliente em broadcast para localizar servidores DHCP disponíveis na rede.
- DHCP Offer: mensagem enviada pelo servidor DHCP oferecendo um endereço IP e demais parâmetros de rede ao cliente.
- DHCP Request: mensagem enviada pelo cliente informando que aceita a oferta recebida e solicitando formalmente a utilização do endereço IP disponibilizado.
- DHCP ACK (Acknowledgement): mensagem enviada pelo servidor confirmando a concessão do endereço IP solicitado e autorizando sua utilização.

A sequência observada durante a captura foi: DHCP Discover -> DHCP Offer -> DHCP Request -> DHCP ACK. A análise dessas mensagens permitiu compreender o funcionamento do protocolo DHCP e verificar o processo de atribuição dinâmica de endereços IP aos dispositivos da rede.

Faça uma alteração no servidor DHCP de modo que ele ofereça endereços IP apenas para as estações de trabalho, cujo MAC Address esteja previamente reconhecido pelo servidor DHCP.
keyboard_arrow_down

Para restringir a distribuição de endereços IP apenas para equipamentos autorizados, foi utilizada a funcionalidade de reserva baseada em endereço MAC. Dessa forma, somente dispositivos previamente cadastrados no arquivo de configuração do servidor DHCP podem receber configurações de rede.

A configuração foi realizada através da inclusão de um bloco no arquivo /etc/dhcp/dhcpd.conf. A diretiva "hardware ethernet" identifica o equipamento através do seu endereço MAC, enquanto a diretiva "fixed-address" define o endereço IP que será atribuído ao dispositivo. Dessa forma, o servidor DHCP reconhece apenas os equipamentos previamente cadastrados e fornece a eles os respectivos endereços IP reservados.

Se colocarmos mais de um servidor DHCP na rede, o que pode acontecer? Faça testes e mostre suas conclusões.
keyboard_arrow_down

Quando dois ou mais servidores DHCP são instalados na mesma rede sem qualquer coordenação entre eles, todos podem responder simultaneamente às mensagens DHCP Discover enviadas pelos clientes. Como consequência, o cliente normalmente aceita a primeira oferta recebida e ignora as demais.

Durante os testes, observou-se que diferentes clientes poderiam receber configurações provenientes de servidores distintos, resultando em inconsistências de endereçamento e parâmetros de rede. As principais consequências observadas são: Distribuição inconsistente de endereços IP; Possíveis conflitos de configuração; Dificuldade de gerenciamento da rede; Problemas de conectividade causados por gateways ou DNS incorretos.
Conclui-se que a utilização de múltiplos servidores DHCP em uma mesma rede deve ser realizada apenas quando existe planejamento adequado, utilizando mecanismos de redundância.

DNS

Qual é o retorno do comando dnsdomainname? O que significa?
keyboard_arrow_down

O comando dnsdomainname exibe o domínio DNS associado ao sistema operacional. Esse domínio é utilizado para complementar nomes de hosts e identificar a qual domínio a máquina pertence. No laboratório realizado, o retorno esperado seria: TerabitSolution.local. Esse resultado indica que a máquina está configurada para utilizar o domínio TerabitSolution.local em suas operações de resolução de nomes.

O que é o nome localhost? E o endereço 127.0.0.1 dado a ele no arquivo /etc/hosts? Por que deve sempre existir este endereço e nome em sistemas UNIX/Linux?
keyboard_arrow_down

O nome localhost representa a própria máquina onde uma aplicação está sendo executada. Ele está associado ao endereço IP 127.0.0.1, conhecido como endereço de loopback.
Esse endereço permite que programas e serviços realizem comunicação interna sem utilizar a interface física de rede. Diversos serviços do sistema operacional dependem dessa configuração para funcionar corretamente, motivo pelo qual a entrada: "127.0.0.1 localhost" deve sempre existir nos sistemas UNIX/Linux.

O que é o FQDN?
keyboard_arrow_down

FQDN (Fully Qualified Domain Name) é o nome completo de um host dentro de um domínio DNS. Ele identifica de forma única um equipamento em uma rede. No laboratório, por exemplo: dns.TerabitSolution.local é um FQDN composto pelo hostname dns e pelo domínio TerabitSolution.local.

Podemos ter 2 servidores DNS na mesma rede? Qual é a configuração mais adequada para esta situação?
keyboard_arrow_down

Sim. É comum existirem dois ou mais servidores DNS na mesma rede para aumentar a disponibilidade do serviço.
A configuração mais adequada consiste em utilizar um servidor DNS primário (master) e um servidor DNS secundário (slave). O servidor secundário mantém uma cópia das zonas DNS do servidor principal e assume as consultas caso o servidor principal fique indisponível. Essa abordagem aumenta a confiabilidade da infraestrutura.

O que é DNS reverso? Como isso foi implementado no lab?
keyboard_arrow_down

O DNS reverso é o processo de resolução que permite obter o nome de um host a partir de seu endereço IP. Enquanto a resolução direta realiza dns.TerabitSolution.local -> 192.168.10.10, a resolução reversa faz o inverso. No laboratório, essa funcionalidade foi implementada através da zona reversa (10.168.192.in-addr.arpa) utilizando registros do tipo PTR.

O que é a entrada MX inserida no domínio? Pode haver mais de uma?
keyboard_arrow_down

O registro MX (Mail Exchange) é responsável por indicar qual servidor receberá as mensagens de correio eletrônico destinadas a um determinado domínio. No laboratório foi configurado a entrada MX apontando para mail.TerabitSolution.local.
Sim, podem existir vários registros MX para um mesmo domínio. Nesse caso, cada registro recebe uma prioridade numérica, permitindo redundância e balanceamento do serviço de correio eletrônico.

O que é resposta autoritativa dada por um servidor DNS? Explique.
keyboard_arrow_down

Uma resposta autoritativa ocorre quando o servidor DNS consultado possui autoridade sobre a zona pesquisada e responde utilizando informações armazenadas diretamente em seus próprios arquivos de configuração. Consultas sobre hosts pertencentes a esse domínio são respondidas diretamente pelo servidor DNS responsável pela zona, sem necessidade de consultar outros servidores (root hints, etc).

O que é um servidor caching-only?
keyboard_arrow_down

Um servidor DNS caching-only é um servidor que não possui zonas autoritativas próprias. Sua função é apenas realizar consultas a outros servidores DNS e armazenar temporariamente os resultados em cache. Quando uma consulta é realizada pela primeira vez, o servidor busca a resposta em servidores externos e as guarda para acelerar requisições das próximas máquinas que buscarem.

SMTP

Descrever as principais modificações realizadas nos arquivos /etc/postfix/main.cf e /etc/postfix/ master.cf e os efeitos de cada modificação. Anexar os arquivos no relatório.
keyboard_arrow_down

O arquivo /etc/postfix/main.cf é o principal arquivo de configuração do Postfix. Durante a configuração do servidor SMTP foram definidos parâmetros relacionados ao nome do servidor, domínio utilizado e redes autorizadas a utilizar o serviço.
As principais configurações utilizadas foram:
myhostname = mail.TerabitSolution.local (define o nome do servidor SMTP)
mydomain = TerabitSolution.local (define o domínio utilizado pelo serviço)
myorigin = $mydomain (define o domínio utilizado como origem das mensagens)
inet_interfaces = all (permite que o serviço aceite conexões em todas as interfaces de rede)
mynetworks = 127.0.0.0/8, 192.168.10.0/24 (define quais redes podem utilizar o servidor para envio de mensagens)
home_mailbox = Maildir/ (define o local de armazenamento das mensagens).

O arquivo /etc/postfix/master.cf controla os serviços executados pelo Postfix e as portas utilizadas pelo servidor. Durante o laboratório foram mantidas as configurações padrão, responsáveis pelo funcionamento do protocolo SMTP.

Qual a diferença entre o armazenamento de e-mails no formato mbox e maildir? Instale o maildir e descreva quais os procedimentos de instalação de um e outro caso.
keyboard_arrow_down

O formato mbox armazena todas as mensagens de um usuário em um único arquivo localizado normalmente em /var/mail/usuario. Já o formato Maildir armazena cada mensagem em arquivos individuais dentro de uma estrutura dividida em diretórios (cur, new, tmp).
Principais diferenças: o mbox agrupa todas as mensagens em um único arquivo, o que o torna mais simples, mas com maior risco de corrupção. O Maildir coloca cada mensagem em um arquivo separado, oferecendo maior segurança, confiabilidade e eficiência para múltiplos acessos.
Para utilizar Maildir foi adicionada a seguinte configuração ao Postfix: `home_mailbox = Maildir/` E para criar a estrutura Maildir para o usuário utilizou-se o formato de subdiretórios no local adequado.

Para que servem os esquemas de autenticação SASL/TLS. Caso tenha instalado, descreva qual foi a solução adotada (Dovecot ou outro) e explique os passos de instalação.
keyboard_arrow_down

O SASL (Simple Authentication and Security Layer) é utilizado para autenticar usuários antes de permitir o envio de mensagens através do servidor SMTP. O TLS (Transport Layer Security) é utilizado para criptografar a comunicação entre cliente e servidor, protegendo credenciais e mensagens durante a transmissão.
A utilização conjunta de SASL e TLS oferece: Autenticação dos usuários, Proteção contra acesso não autorizado, Criptografia das mensagens e Maior segurança do serviço de correio eletrônico. No laboratório foi utilizado o Dovecot para implementação dos serviços POP3 e IMAP, podendo também ser utilizado como mecanismo de autenticação para o Postfix.

Qual é a diferença entre os formatos RFC822 e MIME types definidos para e-mails? Explicá-los.
keyboard_arrow_down

O padrão RFC822 define a estrutura básica de uma mensagem eletrônica, incluindo cabeçalhos (From, To, Date, Subject). Esse formato foi criado para mensagens compostas apenas por texto puro.
O padrão MIME (Multipurpose Internet Mail Extensions) foi desenvolvido para expandir o RFC822, permitindo o envio de: Arquivos anexos, Imagens, Áudio, Vídeo, Conteúdo HTML e Diferentes conjuntos de caracteres. Atualmente praticamente todos os sistemas de correio eletrônico utilizam MIME em conjunto com o formato RFC822.

O que são Server Side Includes (SSI)? Promova alguma alteração no servidor mostrando a funcionalidade desse mecanismo.
keyboard_arrow_down

SSI (Server Side Includes) é um mecanismo que permite ao servidor Web processar determinadas instruções antes de enviar a página ao navegador do cliente. Para demonstrar seu funcionamento foi habilitado o módulo SSI do Apache (módulo include) e criada uma página com extensão .shtml contendo a diretiva `<!--#echo var="DATE_LOCAL" -->`. Quando a página é acessada, o Apache substitui automaticamente a diretiva pela data e hora atuais do servidor, demonstrando o processamento dinâmico realizado no lado do servidor.

Quais são os mecanismos implementados por um servidor Web para conseguir atender tantas conexões simultaneamente? Se houver diretivas de configuração, relacioná-las aqui.
keyboard_arrow_down

Os servidores Web modernos utilizam múltiplos processos, múltiplas threads e mecanismos de concorrência para atender diversos clientes simultaneamente. No Apache, essa funcionalidade é implementada através dos módulos MPM (Multi-Processing Modules), sendo os mais comuns: MPM Prefork, MPM Worker e MPM Event. Esses módulos permitem distribuir as conexões entre processos e threads, aumentando o desempenho e a capacidade de atendimento do servidor de forma robusta e paralela.

Explique qual é a utilidade e como funciona um proxy server no caso do protocolo HTTP.
keyboard_arrow_down

Um proxy HTTP atua como intermediário entre clientes e servidores Web. Em vez de acessar diretamente um site, o cliente envia sua requisição ao proxy, que realiza a consulta ao servidor de destino e devolve a resposta ao usuário. Entre suas principais funções estão: Controle de acesso à Internet, Filtragem de conteúdo, Registro de navegação, Cache de páginas Web, e Melhoria de desempenho e segurança. Em ambientes corporativos, proxies são amplamente utilizados para monitorar e controlar o tráfego Web.

Qual é a relação entre o protocolo MIME e o serviço WEB?
keyboard_arrow_down

O padrão MIME (Multipurpose Internet Mail Extensions) permite identificar o tipo de conteúdo transmitido através dos protocolos da camada de aplicação. No serviço Web, o servidor utiliza MIME Types para informar ao navegador qual é o tipo do arquivo solicitado, enviando dados no cabeçalho HTTP de resposta. Alguns exemplos são: text/html, text/css, image/png, application/pdf. Essas informações permitem que o navegador interprete corretamente o conteúdo recebido para exibição ou download.
